Anatomy of a Python Object at the C level

Every Python object—int, list, a user class—starts with this fixed header in CPython's C code. Click any row to see details.

ob_refcnt
Reference count — how many names point here
ob_refcnt is an integer that tracks how many Python variables (name tags) currently reference this object. Every x = obj increments it; every del x or scope-exit decrements it. When it reaches 0 the object is immediately deallocated — no GC cycle needed. This is Python's primary memory-reclamation mechanism and operates in constant time.
ob_type
Pointer to the type object (e.g. &PyLong_Type)
ob_type is a C pointer to the object's type. The type object stores the object's method table (how to add, print, iterate it, etc.). type(x) in Python simply dereferences this pointer. All instances of int share the same single PyLong_Type object.
ob_size
Item count — only in PyVarObject (list, str, …)
Fixed-size objects (int, float) use plain PyObject. Variable-length objects (list, str, tuple) extend it with ob_size, which stores the number of elements. This is what len(x) reads — an O(1) lookup.
Object Data
The actual payload (integer value, characters, pointers…)
After the header comes the object-specific data. For a Python int this is an array of 30-bit "digits" representing the arbitrary-precision value. For a str it's the UTF-8 / latin-1 / UCS-4 character buffer. For a list it's an array of pointers to other Python objects.
Four-layer allocator hierarchy — click a layer

CPython never calls malloc directly for every object. Instead it routes requests through specialised layers to reduce syscall overhead.

Layer 3 · Object-Specific Allocators
int, float, dict, list — each maintains its own free-list
Types like int and float keep a singly-linked "free-list" of recently deallocated objects so new allocations just pop from the list without touching the OS. CPython caps these lists (e.g. up to 256 freed ints are cached).
↓
Layer 2 · pymalloc (objects ≤ 512 bytes)
Arena → Pool → Block recycling; avoids OS malloc for small objects
pymalloc handles all objects up to 512 bytes. It pre-allocates 256 KB Arenas from the OS, subdivides them into 4 KB Pools, and carves Pools into fixed-size Blocks (8–512 bytes in steps of 8). Freed blocks stay inside their pool and are reused immediately — no free() call to the OS.
↓
Layer 1 · Raw Memory Allocator (objects > 512 bytes)
Thin wrapper over the platform's malloc / realloc / free
Large objects (big strings, large lists) bypass pymalloc and go straight to the platform allocator via PyMem_RawMalloc. This layer exists so CPython can swap in custom allocators (e.g. for debugging or memory tracking) without changing every call site.
↓
Layer 0 · General-Purpose Allocator
C's malloc / free — directly requests memory from the OS
The OS-level allocator (malloc on glibc/Windows CRT). CPython rarely reaches here for short-lived small objects thanks to the layers above. Direct OS calls carry significant overhead — a context switch and potential mmap — which is exactly what pymalloc is designed to avoid.

Pools exist in three states: Used (has at least one free block), Full (all blocks allocated), or Empty (all blocks freed, pool returned to the Arena). CPython keeps a usedpools[64] array — one entry per size class — pointing to the head of a doubly-linked list of Used pools, so allocation is O(1).

512 bytes is the maximum block size. This is size class 64. Any allocation request above 512 bytes exits pymalloc entirely and goes to the Raw Allocator (Layer 1), which calls malloc directly.

Block sizes: 8, 16, 24, … 512 bytes — 64 size classes total. A 14-byte request lands in a 16-byte block (next multiple of 8).

Generational Garbage Collector

Reference counting cannot detect cyclic references (A → B → A). The GC runs periodically to find and break cycles. Click a generation to learn more.

Generation 0 — newest objects
Collected most frequently · threshold: ~700 new allocations since last collection
All newly allocated objects start in Generation 0. The "weak generational hypothesis" states that most objects die young — a local variable in a function lives for microseconds. Generation 0 is collected whenever allocations − deallocations > threshold0 (default 700).
↓ survive a GC cycle → promoted
Generation 1 — survivors
Collected less often · threshold: ~10 Gen-0 collections
Objects that survive one Generation 0 collection are moved here. They are checked every 10 Gen-0 cycles. Module-level variables often live here temporarily before being promoted further if they persist.
↓ survive → promoted
Generation 2 — long-lived objects
Collected rarely · threshold: ~10 Gen-1 collections
Long-lived objects — imported modules, class definitions, global dicts — end up here. A Gen-2 collection is sometimes called a "full GC". It scans everything and can be expensive; Python 3.12+ added incremental collection to reduce pause times.
How cycle detection works: The GC traverses all tracked objects, temporarily decrementing reference counts for internal pointers. Any object whose count reaches 0 after this traversal is unreachable (a cycle) and is collected.
Free-Threaded Python — PEP 703

Removing the GIL requires rethinking every memory primitive. Click each item for details.

pymalloc → mimalloc
Microsoft's per-thread heap allocator — no global lock on allocation
pymalloc relies on the GIL to avoid data races on its shared Arena/Pool structures. Without the GIL every allocation would need a mutex — catastrophically slow. mimalloc uses per-thread heaps: each thread allocates from its own pool with no contention. Cross-thread frees are handled via a lock-free "abandoned" segment list.
Biased Reference Counting
Tracks "local" vs "shared" ref-counts separately per owning thread
Atomic increments/decrements for every single reference change would serialize all threads. Biased RC keeps a fast, non-atomic local count (only the owning thread modifies it) and a slow, atomic shared count (modified by other threads). Most objects are only touched by one thread, so the common path is lock-free.
Deferred & Immortal Reference Counting
Skip counts for hot globals; None / True / False never freed
Deferred RC skips reference counting entirely for objects that are known to be long-lived (top-level functions, built-ins). Instead the GC cleans them up during collection. Immortalization marks objects like None, True, and False with a special sentinel ref-count so they are never freed — removing hot-path atomic ops entirely.
Non-Generational GC
Single-generation collector — no stop-the-world pauses from promotion
In a multi-threaded world, stopping all threads for a GC pause kills scalability. Free-threaded CPython replaces the 3-generation model with a single-generation incremental collector. Objects are not "promoted" — the GC just runs smaller, more frequent incremental passes to keep pause times bounded.
